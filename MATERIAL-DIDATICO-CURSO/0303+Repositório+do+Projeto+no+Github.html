Instruções Básicas de Git

Git é o sistema de controle de versão mais utilizado no mundo. Ele permite que você rastreie mudanças no código, colabore com outros desenvolvedores e mantenha um histórico completo de todas as alterações feitas em um projeto. Se você ainda não domina Git, não se preocupe - vamos cobrir os comandos essenciais que você precisa conhecer.

1. Clonando o Repositório na Sua Máquina

Clonar um repositório significa fazer uma cópia completa dele (incluindo todo o histórico de commits) na sua máquina local. Esse é sempre o primeiro passo para começar a trabalhar em um projeto Git.

🔽 Como Clonar o Repositório

Abra o terminal (Linux/macOS) ou Git Bash (Windows) e navegue até a pasta onde você deseja armazenar o projeto. Em seguida, execute:

git clone https://github.com/leandrocgsi/rest-with-spring-boot-and-java-erudio.git

Esse comando irá criar uma nova pasta chamada automated-tests-with-java-erudio contendo todo o conteúdo do repositório. Você verá uma saída similar a esta:

Cloning into 'automated-tests-with-java-erudio'... remote: Enumerating objects: 1543, done. remote: Counting objects: 100% (1543/1543), done. remote: Compressing objects: 100% (892/892), done. remote: Total 1543 (delta 456), reused 1401 (delta 314) Receiving objects: 100% (1543/1543), 2.45 MiB | 3.21 MiB/s, done. Resolving deltas: 100% (456/456), done.

💡 Dica: Se você não tem Git instalado na sua máquina, baixe-o em git-scm.com/downloads. A instalação é simples e rápida em qualquer sistema operacional.

2. Importando o Projeto na IntelliJ IDEA

Depois de clonar o repositório, o próximo passo é importá-lo na IntelliJ IDEA para começar a trabalhar no código. A IntelliJ é uma das IDEs mais poderosas para desenvolvimento Java, com excelente suporte para Maven, Spring Boot e testes automatizados.

📂 Passos para Importar na IntelliJ

  1. Abra a IntelliJ IDEA
  2. Clique em File → Open (ou Open na tela inicial)
  3. Navegue até a pasta automated-tests-with-java-erudio que você acabou de clonar
  4. Selecione a pasta raiz do projeto e clique em OK
  5. A IntelliJ detectará automaticamente que é um projeto Maven e perguntará se deseja importá-lo
  6. Confirme a importação e aguarde enquanto a IDE indexa os arquivos e baixa as dependências

Durante a importação, você verá uma barra de progresso na parte inferior da IDE indicando "Indexing..." e "Resolving Maven dependencies...". Esse processo pode levar alguns minutos na primeira vez, dependendo da sua conexão com a internet e da velocidade do seu computador.

⚠️ Importante: Certifique-se de ter o JDK correto configurado na IntelliJ. O projeto usa Java 17 ou superior. Você pode verificar e configurar o JDK em File → Project Structure → Project SDK.

3. Baixando Mudanças do Repositório (Pull)

Conforme o curso avança e novos exemplos são adicionados ao repositório, você precisará atualizar sua cópia local para ter acesso às versões mais recentes do código. O comando git pull faz exatamente isso - ele baixa as mudanças do repositório remoto e as mescla com seu branch local.

🔄 Atualizando Seu Repositório Local

Via Terminal:

# Certifique-se de estar na pasta do projeto cd automated-tests-with-java-erudio # Baixe as últimas mudanças git pull origin main

A saída será similar a:

remote: Enumerating objects: 15, done. remote: Counting objects: 100% (15/15), done. remote: Compressing objects: 100% (8/8), done. remote: Total 10 (delta 4), reused 8 (delta 2) Unpacking objects: 100% (10/10), done. From https://github.com/leandrocgsi/rest-with-spring-boot-and-java-erudio a1b2c3d..e4f5g6h main -> origin/main Updating a1b2c3d..e4f5g6h Fast-forward src/main/java/NewClass.java | 25 +++++++++++++++++++ 1 file changed, 25 insertions(+)

💻 Via IntelliJ IDEA

A IntelliJ oferece uma interface gráfica conveniente para operações Git:

  1. Pressione Ctrl+T (Windows/Linux) ou Cmd+T (macOS)
  2. Ou vá em VCS → Git → Pull
  3. Na janela que abrir, confirme o branch (deve ser main) e clique em Pull

💡 Boa Prática: Sempre execute git pull antes de começar a trabalhar em algo novo. Isso garante que você está trabalhando com a versão mais atualizada do código e minimiza conflitos de merge.

4. Fazendo Commit das Suas Alterações

Depois de fazer modificações no código - seja implementando novos testes, corrigindo bugs ou adicionando funcionalidades - você precisa "commitar" essas mudanças. Um commit é como um snapshot do seu projeto em um determinado momento, com uma mensagem descritiva explicando o que foi alterado.

📝 Criando Commits via Terminal

O fluxo básico envolve três etapas:

# 1. Adicionar arquivos modificados à área de staging git add . # Ou adicionar arquivos específicos git add src/test/java/MeuNovoTest.java # 2. Criar o commit com uma mensagem descritiva git commit -m "Implementa testes unitários para a classe SimpleMath" # 3. Verificar o status (opcional mas recomendado) git status

A mensagem do commit deve ser clara e descrever o que foi feito e, preferencialmente, por quê. Exemplos de boas mensagens:

✅ Boas Mensagens de Commit:

  • "Adiciona validação de entrada nula no método divide"
  • "Corrige bug no cálculo de média quando lista está vazia"
  • "Refatora testes de integração para usar TestContainers"
  • "Implementa testes parametrizados para validação de CPF"

⚠️ Evite Mensagens Genéricas:

  • "Update" ❌
  • "Fix" ❌
  • "Mudanças" ❌
  • "asdf" ❌

Mensagens vagas dificultam entender o histórico do projeto posteriormente.

💻 Criando Commits via IntelliJ IDEA

A IntelliJ torna o processo de commit muito mais visual:

  1. Pressione Ctrl+K (Windows/Linux) ou Cmd+K (macOS)
  2. Ou vá em VCS → Commit
  3. Na janela de commit, você verá todos os arquivos modificados
  4. Marque os arquivos que deseja incluir no commit
  5. Escreva uma mensagem de commit descritiva no campo de texto
  6. Revise as mudanças clicando em cada arquivo (diff será exibido)
  7. Clique em Commit ou Commit and Push

💡 Dica Profissional: Use "Commit and Push" quando tiver certeza de que suas mudanças estão corretas e deseja enviá-las imediatamente ao repositório remoto. Caso contrário, use apenas "Commit" para salvar localmente e depois fazer push manualmente.

5. Enviando Mudanças para o GitHub (Push)

Commits são salvos apenas localmente na sua máquina. Para compartilhar suas mudanças com outros desenvolvedores ou fazer backup no repositório remoto (GitHub), você precisa fazer "push".

☁️ Enviando Commits via Terminal

# Enviar commits do branch atual para o repositório remoto git push origin main # Se for seu primeiro push em um novo branch git push -u origin nome-do-branch

Após executar o push, você verá algo como:

Enumerating objects: 8, done. Counting objects: 100% (8/8), done. Delta compression using up to 8 threads Compressing objects: 100% (4/4), done. Writing objects: 100% (5/5), 612 bytes | 612.00 KiB/s, done. Total 5 (delta 2), reused 0 (delta 0) To https://github.com/leandrocgsi/rest-with-spring-boot-and-java-erudio.git a1b2c3d..e4f5g6h main -> main

💻 Fazendo Push via IntelliJ IDEA

  1. Pressione Ctrl+Shift+K (Windows/Linux) ou Cmd+Shift+K (macOS)
  2. Ou vá em VCS → Git → Push
  3. Revise os commits que serão enviados
  4. Clique em Push

⚠️ Atenção ao Push: Antes de fazer push, certifique-se de que:

  • Seu código compila sem erros
  • Todos os testes passam
  • Você fez pull das últimas mudanças (para evitar conflitos)
  • Não está enviando arquivos sensíveis (senhas, chaves API, etc.)

Troubleshooting: Problemas Comuns e Soluções

Durante a configuração do ambiente e uso do Git, alguns problemas são bastante comuns. Vamos abordar os principais e suas soluções.

Problema 1: "git: command not found"

⚠️ Sintoma: Ao tentar executar comandos Git, você recebe a mensagem "command not found" ou "não é reconhecido como comando interno ou externo".

Solução:

  • Verifique se o Git está instalado: baixe em git-scm.com
  • No Windows, certifique-se de que o Git foi adicionado ao PATH durante a instalação
  • Após instalar, feche e reabra o terminal para que as mudanças tenham efeito

Problema 2: Conflitos de Merge

⚠️ Sintoma: Ao fazer git pull, você vê mensagens como "CONFLICT (content): Merge conflict in arquivo.java".

Solução:

  1. Abra os arquivos em conflito na IntelliJ (ela os marcará automaticamente)
  2. Você verá marcadores como <<<<<<<, ======= e >>>>>>>
  3. Use a interface de merge da IntelliJ: clique em Accept Yours, Accept Theirs ou edite manualmente
  4. Após resolver, adicione o arquivo: git add arquivo.java
  5. Complete o merge: git commit

Problema 3: IntelliJ Não Reconhece o Projeto Maven

⚠️ Sintoma: Após importar o projeto, a IntelliJ não baixa as dependências ou não reconhece a estrutura Maven.

Solução:

  • Clique com botão direito no arquivo pom.xml
  • Selecione Maven → Reload Project
  • Se o problema persistir, vá em File → Invalidate Caches / Restart
  • Verifique se há erros no painel "Maven" (View → Tool Windows → Maven)

Problema 4: Erro de Autenticação no Push

⚠️ Sintoma: Ao tentar fazer push, você recebe "Authentication failed" ou "Permission denied".

Solução:

  • Desde agosto de 2021, GitHub não aceita mais senha para autenticação via HTTPS
  • Você precisa configurar um Personal Access Token (PAT)
  • Vá em GitHub → Settings → Developer settings → Personal access tokens → Generate new token
  • Use o token gerado como senha ao fazer push
  • Ou configure SSH keys para autenticação mais conveniente

Boas Práticas e Workflow Profissional

Agora que você sabe os comandos básicos, vamos discutir como usar Git de forma profissional em um projeto real.

✅ Ciclo de Trabalho Recomendado

  1. Antes de começar: git pull origin main
  2. Durante o trabalho: Faça commits frequentes com mensagens claras
  3. Antes de commitar: Teste seu código localmente
  4. Ao commitar: Use mensagens descritivas no presente/imperativo
  5. Antes de fazer push: Faça pull novamente para garantir que está atualizado
  6. Ao fazer push: Verifique que todos os testes passam

✅ Estrutura de Mensagens de Commit

Um padrão comum é usar prefixos para categorizar commits:

  • feat: Nova funcionalidade
  • fix: Correção de bug
  • test: Adição ou modificação de testes
  • refactor: Refatoração de código existente
  • docs: Documentação
  • style: Formatação, ponto e vírgula faltando, etc.

Exemplo: feat: implementa validação de CPF com testes unitários

💡 Git Flow Simplificado:

Para projetos maiores, considere usar branches separados para diferentes funcionalidades. O fluxo básico seria:

  1. Criar um branch para sua funcionalidade: git checkout -b feat/nova-funcionalidade
  2. Fazer commits nesse branch
  3. Quando estiver pronto, fazer merge de volta ao main
  4. Isso mantém o main sempre estável
